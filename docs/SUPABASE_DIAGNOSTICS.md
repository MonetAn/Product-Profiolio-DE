# Диагностика проблем с Supabase (pending, не работает локально)

Если приложение поднимается локально, но запросы к Supabase висят в **pending** в Network, по шагам проверь следующее.

## 1. Какой бэкенд используется

В `.env` указан **хостинг Supabase** (`https://xxx.supabase.co`), а не локальный Docker. То есть фронт на localhost обращается к облаку.

- Если хочешь локальный Supabase: установи [Supabase CLI](https://supabase.com/docs/guides/cli), выполни `supabase start` и в `.env` подставь URL и anon key из `supabase status`.
- Если используешь только хостинг — все проверки ниже про облачный проект.

## 2. Проект на паузе (Free tier)

На бесплатном тарифе проект **засыпает** после неактивности. Первые запросы после паузы могут идти 30–60+ секунд (cold start).

- Зайди в [Supabase Dashboard](https://supabase.com/dashboard) → свой проект.
- Если проект «paused» — нажми **Restore project**. Подожди 1–2 минуты и обнови приложение.

## 3. Сеть и DNS

- **VPN / корпоративный прокси** — отключи и проверь снова.
- В DevTools → Network: посмотри, запросы к `*.supabase.co` уходят (Initiator, URL) или сразу pending. Если статус так и не меняется — возможна блокировка или таймаут на стороне сети.
- В консоли браузера при старте приложения в dev выводится результат проверки подключения к Supabase (см. код в `client.ts`). Если там ошибка или таймаут — проблема доступа к URL.

## 4. Поведение в браузере

- **Один раз открой приложение** и дождись ответа (до 1–2 минут при cold start). Не обновляй страницу и не переключай вкладки в момент первых запросов.
- Проверь **Console**: есть ли красные ошибки (CORS, 401, 5xx)? Есть ли предупреждения `[useAccess]` или `[Supabase]`?
- В **Network** отфильтруй по Fetch/XHR: несколько одинаковых `get_my_access` или `initiatives` могут быть из-за повторного рендера/Strict Mode. Важно: хотя бы один запрос должен завершиться (200 или 4xx). Если все висят — см. п. 2 и 3.

## 5. Проверка без приложения

Проверь доступность проекта с той же машины:

```bash
# Подставь свой URL из .env (VITE_SUPABASE_URL)
curl -s -o /dev/null -w "%{http_code}\n" "https://ТВОЙ_PROJECT_REF.supabase.co/rest/v1/"
```

Ожидаемо: `200`, `401` или `404` — значит, до бэкенда доходим. Таймаут или ошибка DNS — сетевая проблема.

## 6. Ключи и переменные

- В `.env` должны быть `VITE_SUPABASE_URL` и `VITE_SUPABASE_PUBLISHABLE_KEY` (anon key из Dashboard → Settings → API).
- После смены `.env` **перезапусти dev-сервер** (`npm run dev`), иначе старые значения останутся в бандле.

## 7. Если используешь локальный Supabase (Docker)

- `supabase status` — все сервисы должны быть `running`.
- `supabase stop` затем `supabase start` — перезапуск при «странном» состоянии.
- Логи: `docker compose logs postgres` или `supabase logs` — смотреть на ошибки при запросах.

---

**Кратко:** чаще всего «всё висит в pending» — это **приложение снова открыли после паузы проекта** (Free tier) или сетевая задержка/блокировка. Сначала проверь Dashboard (Restore) и один раз подожди 1–2 минуты без обновления страницы.
