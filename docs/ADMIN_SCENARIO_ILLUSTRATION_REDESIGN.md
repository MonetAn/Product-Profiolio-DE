# Перезапуск: иллюстрация «Заполни по шагам»

## 1. Почему текущая реализация не сработала (технический разбор)

### 1.1 Симптом

На экране — **одна крошечная оранжевая точка** слева; тропа из следов не читается. По hover каскад стал длиннее (задержки 300 ms × 12), но **масштаб** остался неверным.

### 1.2 Вероятные причины (что не учли)

| Фактор | Суть |
|--------|------|
| **Цепочка flex + `min-h-0`** | Карточка в grid с `min-h-0`, кнопка `h-full`, колонка flex. Дочерний блок с SVG **может сжиматься** по главной оси/поперечной, если не зафиксирована **жёсткая зона** под иллюстрацию. Итог: у SVG остаётся микроскопическая **эффективная область** — `meet` масштабирует всю сцену под неё → всё превращается в «точку». |
| **Один flex-ряд с текстом** | Иллюстрация, заголовок и подзаголовок в **одной** колонке без выделенной строки grid. При вертикальном центрировании (`justify-center`) группа контента по центру экрана; при нехватке высоты **первый flex-элемент** (иллюстрация) часто **первым теряет место**, если у него нет `flex-shrink: 0` и явного **минимума в px**. |
| **SVG `preserveAspectRatio="meet"`** | Сцена 480×108 в прямоугольнике с **непредсказуемым** соотношением сторон. Если фактическая высота полосы под SVG ~10–20 px (из-за сжатия), масштаб по высоте убивает размер; если ширина коллапсирует — то же по ширине. |
| **Визуальный контраст** | Даже при верном масштабе серые следы на сером фоне карточки **плохо различимы**; заметна только первая **оранжевая** (hover) — усиливает ощущение «одной точки». |
| **Сложность сцены** | 12 отдельных `transform` + path в одном viewBox увеличивают риск обрезки и ошибок позиционирования; отладка «почему мелко» без инспектора сложна. |

### 1.3 Вывод для новой версии

Нельзя полагаться на «SVG растянется сам» внутри глубоко вложенного flex/grid. Нужно одно из:

- **Якорь в px**: зона иллюстрации с `min-height: 140–180px` (или больше), `flex-shrink: 0`, при необходимости **отдельная строка** layout (grid `auto 1fr` или фиксированный верхний блок).
- Либо **не SVG как единственный носитель масштаба**: крупные элементы в **px/rem** (div + border-radius, или несколько простых shape), либо фоновый паттерн + маска.
- Сначала **доказать размер** (один большой прямоугольник/круг на всю ширину зоны), потом заменить на следы.

## 2. Лучшие практики (кратко, с опорой на веб)

1. **Flexbox / SVG** ([SO: stretch SVG in flexbox](https://stackoverflow.com/questions/44175663/stretch-svg-in-flexbox-to-max-height-keeping-aspect-ratio)): явные размеры на SVG и родителе, `min-height` на контейнере, при необходимости **grid** вместо flex для стабильных пропорций карточки.
2. **Микроанимации** ([micro-interactions guide](https://www.frontendtools.tech/blog/micro-interactions-ui-ux-guide)): 100–300 ms для отклика; **смысл** анимации — подтверждение и направление взгляда, не шоу.
3. **Дашборды** ([motion for dashboards](https://themotiondot.com/blog/motion-design-for-data-dashboards-7-practical-patterns)): стабильные якоря, «слои» раскрытия — сначала главное, без перегруза.
4. **Доступность**: `prefers-reduced-motion` — статичная или упрощённая версия.

## 3. Мнение UX/UI дизайнера (независимый план «красиво и по-додовски»)

*Ниже — предложение без привязки к «строго слева направо» и без требования именно 12 следов.*

### 3.1 Иерархия

Карточка выбора сценария — **решение за 2–3 секунды**. Визуал должен:

1. Мгновенно отличать карточку «пошагово» от «таблица».
2. Не спорить с заголовком за внимание.

**Рекомендация:** выделить **верхнюю треть** карточки под **одну сильную визуальную идею** (полоса 120–160 px высотой), ниже — заголовок и подстрочник. Не смешивать мелкие детали и типографику в одной зоне.

### 3.2 Характер бренда (Dodo)

- Тёплый, человечный, слегка ироничный тон.
- **Крупные простые формы**, не «инженерная» схема.
- Оранжевый — **акцент**, не фон всей графики.

### 3.3 Концепции (на выбор)

**A. «Тропа как фон-полоса»**  
Широкая мягкая полоса (градиент или очень светлый orange tint) с **3–5 крупными** следами (каждый 48–72 px в экранных px), диагональ или дуга **от угла к углу** блока. При hover — мягкое **подсветка полосы** или поочерёдное **лёгкое увеличение** следов (scale 1.05), не обязательно 12 шагов.

**B. «Один персонаж + след»**  
Упрощённый силуэт/иконка «шага» (два следа крупно) + короткий текстовый намёк в анимации. Меньше элементов — **выше узнаваемость**.

**C. «Прогресс-нarrative»**  
Тонкая линия-путь с **4–6 крупными точками/следами**; hover заполняет их как прогресс. Связь с «шагами» очевидна, анимация короче по времени.

### 3.4 Анимация

- **Длительность** одного «шага» визуала: 200–350 ms, полный hover-цикл **не длиннее ~2 s**, иначе раздражает при случайных наведениях.
- **Easing**: ease-out.
- На клавиатурном фокусе — тот же эффект, что на hover.

### 3.5 Итог дизайнера

Приоритет: **крупность и узнаваемость одним взглядом**, затем **лёгкий живой отклик**. Двенадцать мелких следов в узкой полосе — хрупкий паттерн; **3–6 очень крупных** или **одна выразительная композиция** надёжнее для UI.

## 4. План полного перезапуска (реализация)

### Фаза 0 — Демонтаж

- Удалить из `index.css` все правила `.admin-scenario-bird-foot`, `.admin-scenario-quick:hover … foot`, связанные с `.admin-scenario-footprints-*`.
- Упростить или удалить `ScenarioFootstepsIllustration` в `AdminScenarioIllustrations.tsx` (или файл целиком переписать).
- В `Admin.tsx`: убрать класс `admin-scenario-quick` с кнопки до появления новой логики (или заменить на новый BEM-префикс).
- Вернуть при необходимости `overflow-hidden` на обёртке карточки, если визуал снова вписан внутрь.

### Фаза 1 — Каркас layout (обязательно до графики)

- Разбить левую карточку на зоны, например:
  - **Ряд 1:** `min-height: 140px` (или `clamp(120px, 18vh, 180px)`), `width: 100%`, `flex-shrink: 0`, внутренние отступы согласованы с карточкой.
  - **Ряд 2:** заголовок.
  - **Ряд 3:** подзаголовок.
- Проверка: в ряд 1 вставить **временный** `div` с `background: orange/20`, `min-height: 140px` — он должен **занимать всю ширину** карточки и быть хорошо виден. Если нет — чинить только layout.

### Фаза 2 — Визуал (новая графика)

**Вариант 2a (рекомендуемый): HTML/CSS следы**

- 4–6 блоков «след»: комбинация `border-radius` + `transform: rotate()` или небольшой inline-SVG **на каждый след** с фиксированными `width`/`height` в **rem/px** (например 3rem × 3.5rem).
- Расположение: диагональ или дуга **от одного края полосы к другому** (не обязательно строго горизонталь).
- Цвет покоя: `foreground/15` + `border`; hover: `primary` с задержкой по `nth-child` или `animation`.

**Вариант 2b: один широкий SVG**

- SVG только внутри зоны фазы 1; атрибуты `width="100%"` `height="100%"` и **`preserveAspectRatio="xMidYMid meet"`** при **фиксированной высоте родителя** в px.
- Внутри viewBox — **немного** крупных фигур (не 12 мелких).

### Фаза 3 — Анимация

- CSS: `transition` на `fill`/`background`/`transform` + `transition-delay` по индексу; или короткий `@keyframes` «pulse» по очереди.
- `prefers-reduced-motion`: все следы в спокойном акцентном состоянии без каскада.

### Фаза 4 — Проверка

- Chrome/Safari/Firefox, ширина карточки ~320px и ~800px, тёмная тема.
- Масштаб страницы 100% и 125%.
- Клавиатура: focus-visible = тот же эффект, что hover.

### Фаза 5 — Правая карточка (по желанию)

- Выровнять **визуальный вес** с левой (та же высота верхней полосы, своя метафора «таблица»).

## 5. Резюме

| Было | Станет |
|------|--------|
| SVG на всю ширину без гарантированной высоты зоны | Выделенная полоса с **min-height в px** и `flex-shrink: 0` |
| 12 мелких следов в одном viewBox | **4–6 крупных** следов в px/rem или упрощённая композиция |
| Слабый контраст в покое | Явная обводка/фон следа |
| Долгий hover-цикл | Укоротить или оставить мягкий каскад ≤ ~2 s |

После утверждения этого плана следующий шаг — **Фаза 0 + Фаза 1** в коде, затем графика по **2a** или **2b**.

## 6. План v3: птичьи (додо-подобные) лапы + тропа «идут куда-то»

### 6.1 Ревью запроса (UX/UI)

| Вопрос | Ответ дизайнера |
|--------|------------------|
| **Текущие лапы уже хороши?** | Да: крупно, читаемо, мягкая анимация. Абстрактная «подушечка + три точки» универсальна. |
| **Станут ли «лучше» именно птичьи лапы?** | **Иначе, не обязательно лучше.** Птичий силуэт (плавник между пальцами) усиливает **бренд и узнаваемость** («это про Додо/птицу»), но добавляет деталей. Оптимум — **упрощённый** веб-пальцевый след в том же духе (крупный, мягкие формы). |
| **Совместить стиль и птичьи лапы?** | Да: сохранить **ту же палитру, контраст покоя, каскад hover ~280 ms, фиксированную полосу**, заменить только **форму** на SVG-силуэт утиного/птичьего следа. |
| **Не в ряд, а «идут куда-то» — реалистично?** | **Да, очень.** Ряд читается как «инструкция: шаг 1–5». **Дуга / диагональ вверх-вправо** читается как **движение к цели** (к заголовку, к действию), без ощущения чеклиста. В стоках часто используют [тропу по кривой](https://www.vecteezy.com/vector-art/1266339-footprints-on-a-curved-path) и [следы водоплавающих птиц](https://www.vecteezy.com/vector-art/29376450-path-of-duck-goose-swan-footprints-tracks-of-paw-prints-of-seagull-cormorant-pelican-flamingo-albatross-silhouette-trail-of-webbed-footed-birds-black-vector-isolated-on-white-print-game). |

### 6.2 Что искали в интернете (и зачем)

1. **Силуэт webbed footprint** — Vecteezy / Freepik «duck footprint», «path of duck swan footprints» — форма **трёх пальцев + перепонка**, минимальный силуэт.
2. **Кривая тропа** — «footprints curved path», «footprint trail perspective» — идея **не линейного** расположения, направление «вглубь» или по дуге.
3. **Микроанимация** — уже в §2: каскад **≤ ~1.7 s** на 6 шагов.

### 6.3 Реализация (технически)

1. **Контейнер** — как сейчас: `relative`, **фиксированная высота** (~168–184px), `flex-shrink: 0`, `w-full`.
2. **6 следов** — `position: absolute`, координаты **left % / bottom %** по **лёгкой дуге** снизу-слева → вверх-вправо (к зоне заголовка). Чередование **левой/правой** лапы: `scaleX(-1)` + разный наклон.
3. **Графика** — один **inline SVG** на след: `viewBox` фиксированный, размер **w-14 h-16** (sm чуть крупнее), класс `.admin-scenario-dodo-foot` на `<path>` (fill + stroke, transition).
4. **Hover** — те же задержки **0, 280, … ms** по `nth-child` на обёртках следов внутри `.admin-scenario-dodo-trail`.
5. **Reduced motion** — все следы в спокойном primary без каскада.

### 6.4 Критерии готовности

- Следы читаются как **птичьи** с первого взгляда.
- Тропа **не горизонтальная линейка** — видна **траектория** «в сторону» контента.
- Размер и контраст не хуже текущей v2.

## 7. v4–v7

- **Текущее:** **5 следов**, снова **HTML**: три **отдельных** кружка-пальца + **ступня**; между ними **зазор** (не `-mb` внахлёст); пальцы **светлее** и с **обводкой цвета карточки**, ступня **плотнее** с **прямым верхом** — не сливаются. Каскад **280ms** на `.admin-scenario-step-print` + pad/toe.
- **Траектория:** следы не в одну линию — **absolute** `left%` / `bottom%` по **дуге** снизу-слева вверх-вправо, зеркало лапы и лёгкий **scale** на шаг для ощущения «идущей» тропы; зона **фиксированной высоты** + `overflow-hidden` + отступы, чтобы не заезжать на заголовок.
- **Hover:** не сплошной primary — **градиент ~12–22%** primary, **обводка** primary, мягкий shadow и **scale ~1.04** (без «плоского оранжевого блока» на всей карточке).
- **Правая карточка:** сетка **5×3**, крупнее ячейки; hover — primary + задержки **90×row + 42×col ms**, `scale(1.04)`, easing помягче. Reduced motion: статичная сетка primary, без scale.

## 7b. v4–v6 (архив)

- **Откат SVG:** три толстых округлых пальца + подушечка; **4 следа**, поворот **84–96°**; размеры **5.5×6.5rem** / **8×7.75rem** на `sm`.
- **v6:** уплощённые три бугра + 90° — заменено на вариант выше.
- **v5:** четыре пальца — откат из-за чтения жеста.
- **Карточки:** по-прежнему симметричные `flex-1` сверху/снизу + зазор до заголовка.

## 7a. Ранее v4 — «плюшевые» три пальца + подушечка

- **Форма:** один замкнутый path (viewBox 64×76) — три широких округлых «пальца» + общая подушечка, без острых углов.
- **Композиция:** **4 крупных** следа в ряд (`~5.5–8rem`), направление **вправо**, чередование высоты (зигзаг) и зеркало лапы (`scaleX(±1)`).
- **Покой:** приглушённая заливка `muted`, тонкая обводка, мягкий `drop-shadow` (в тёмной теме — чуть сильнее контраст).
- **Hover:** по очереди с шагом **280 ms** — `primary`, двойное свечение (узкое + широкое blur), лёгкий **scale ~1.055** на SVG; `focus-visible` = тот же эффект.
- **Reduced motion:** на карточке без каскада — статичные следы в `primary`; в CSS отключены transition/scale на тропе при `prefers-reduced-motion`.
- **Сетка карточек (до v5):** фиксированная высота верхней зоны; в **v5** заменено на `flex-1` сверху + общий `min-h` зоны иллюстрации у обеих карточек.
