# План: заполнение по шагам (quick flow) v2

## Цели

1. **Подтверждение контекста** — юнит/команда из профиля показываются явно, пользователь подтверждает или меняет.
2. **Мульти-селект команд** — можно выбрать несколько команд; порядок = порядок прохождения.
3. **Только юнит в профиле** — команды не подставляем; обязательный выбор ≥1 команды.
4. **Последовательность** — полный флоу (шаг 1 → шаг 2) для **одной** команды, затем «к следующей команде».
5. **Полноэкранный онбординг** вместо модалки с затемнением.

---

## Поток экранов

### Экран A — старт (без изменений)

Две карточки: «Заполни по шагам» | «Открой всю таблицу».

### Экран B — полноэкранная настройка (новый)

Занимает область контента под хедером админки (не `Dialog`).

**B1. Юнит**

| Условие | UI |
|---------|-----|
| В профиле есть `member_unit` | Карточка: «Ваш юнит: **{name}**» → «Да, верно» / «Выбрать другой» |
| Нет юнита | Выпадающий список юнитов |

После подтверждения юнита → **B2**.

**B2. Команды (мульти-селект)**

| Условие | UI |
|---------|-----|
| В профиле есть `member_team` (и выбранный юнит совпадает) | Чекбоксы команд юнита; **профильная команда отмечена по умолчанию**; можно отметить ещё |
| Только юнит, команды в профиле нет | Тот же список; **ничего не выбрано**; текст: «Выберите одну или несколько команд. По каждой мы пройдём шаги отдельно.» |

Кнопка **«Начать заполнение»** активна при ≥1 выбранной команде.

Вторичная: **«Назад»** → возврат на экран A без смены URL режима quick.

### Экран C — текущий quick flow (шаги 1–2)

- В шапке блока: **«Команда 2 из 4 · {TeamName}»** + при необходимости юнит мелким текстом.
- Логика шагов без изменений по смыслу.
- **Шаг 2**, если в очереди ещё есть команды: первичная кнопка **«Сохранить и перейти к следующей команде»** (делает save + сброс шага на 1 + смена фильтра на следующую команду).
- **Последняя команда в очереди**: **«Сохранить и завершить»** → сохранение, очистка очереди, выход из `mode=quick`, возврат на старт выбора сценария.

## Техническая модель

- **URL**: `mode=quick&units={U}&teams={currentTeam}` — в один момент времени в quick-режиме ровно **одна** команда (как сейчас для фильтрации данных).
- **Очередь**: `sessionStorage` ключ `admin_quick_team_queue_v1`, значение `{ unit, teams: string[], currentIndex: number }`.
- При старте: `teams = выбранный порядок`, `currentIndex = 0`, URL `teams = teams[0]`.
- При «следующая команда»: сохранить черновик → `currentIndex++` → URL `teams = teams[currentIndex]` → `quickStep = 1` → очистить локальные патчи и `createdInQuickSession`.
- При «завершить» или выходе: удалить ключ из sessionStorage.

Ограничение: при открытии в новой вкладке без session очередь теряется — приемлемо; в подсказке можно не афишировать.

## Senior UX/UI review (кратко)

| Тема | Вердикт |
|------|---------|
| Полный экран вместо модалки | **Да.** Меньше клаустрофобии, легче читать длинные названия юнитов/команд, проще мульти-селект. |
| Подтверждение профиля | **Да.** Снижает ошибочный ввод в чужой юнит. |
| Одна команда в фокусе в флоу | **Да.** Сумма 100% и валидация привязаны к команде; смешивание в одном шаге запутало бы. |
| Явный прогресс «2 из 4» | **Обязательно.** Снижает тревогу «сколько ещё». |
| Сохранение перед сменой команды | **Да.** Иначе потеря данных; одна кнопка «Сохранить и дальше» лучше двух шагов. |
| Назад с экрана B | **На старт сценариев** — ожидаемо. |
| Визуальная иерархия B | Крупный заголовок «Заполнение по шагам», подзаголовок про кварталы; карточки с лёгкой тенью/бордером; CTA один основной внизу sticky на мобиле (опционально). |

**Риски:** длинный список команд — ограничить высоту списка с прокруткой + счётчик «Выбрано: N».
