## 1. Правила и рамки (детально)

### 1.1 Область применения (IN SCOPE)

| Что входит | Пояснение |
|------------|-----------|
| Все экраны **админки** | Страницы под маршрутом/оболочкой админа: выбор сценария, заполнение, люди, доступ, активность, диалоги и компоненты из `src/components/admin/`, `src/pages/Admin*.tsx`, обёртка `AdminRoute` / общий layout админки. |
| Навигация и хедер админки | Табы, «Дашборд», бейджи («Сохранено», счётчики), настройки в контексте админки. |
| Иллюстрации и иконки **только в админке** | SVG/PNG в сценариях, пустые состояния админки, декор карточек выбора. |

### 1.2 Вне области (OUT OF SCOPE — не трогать без отдельного решения)

| Что не трогаем | Пояснение |
|----------------|-----------|
| **Графики** | Любые чарты/визуализации данных вне админки или явно помеченные как «дашборд-метрики» — цвета и стили **как сейчас**, пока нет отдельного ТЗ. |
| **Публичный дашборд / портфель** | Не перекрашивать под DE только из-за админки. |
| **UX и сценарии** | Не менять порядок шагов, полей, логику сохранения, структуру табов — только визуал. Исключение: микро-правки доступности (контраст фокуса) — по согласованию. |
| **API и данные** | Редизайн не меняет контракты и бизнес-логику. |

### 1.3 Выбранный визуальный режим

- **Активный вариант: A — светлая админка + палитра DE:** светлый фон и карточки, акцент **индиго** `#694BFB`, шрифт Juneau в админке; без оранжевого Brands как primary.
- **Вариант B (тёмный shell):** отложен / не используется после валидации; при желании можно вернуться отдельным решением.

### 1.4 Бренд Dodo Engineering (правила из референсов)

1. **Палитра:** использовать градации из гайда (тёмный / основной / светлый) для акцента и состояний; оранжевый столбец Brands **не** как primary в админке.
2. **Градиенты:** в основном для **фона** (полушария, мягкое свечение); UI-элементы (кнопки, бейджи) — сплошные заливки или обводки, не «градиентный текст».
3. **Типографика:** в админке **Inter** — основной текст и UI; **Juneau** — опционально на крупных заголовках (`font-juneau font-medium`: «Админка», названия сценариев, ключевые h1/h2). Fallback Helvetica по гайду DE.
4. **Иконки:** линейные, тонкие (ориентир 2px мелкие, 3px крупные); при необходимости 3D-иконки — точечно, не массово.
5. **Линии:** тонкие лучи/направления — аксессуар, не доминируют.
6. **Blur / стекло:** по гайду — без скруглений у слоя blur; использовать выборочно (оверлеи, панели).

### 1.5 Критерии качества

- Контраст текста и интерактивов — читаемость и фокус (WCAG по возможности для ключевых состояний).
- `prefers-reduced-motion`: не усложнять анимации; существующие упрощения сохранять.
- Единый источник цветов: CSS variables / тема **админки**, без разброса «магических» hex в десятках файлов (по мере рефакторинга).

### 1.6 Документы-спутники

- `docs/ADMIN_UI_COLORS_AND_THEME.md` — **тема админки:** класс `admin-engineering`, индиго primary, голубой для «внимания», отделение от оранжевого дашборда.
- `docs/ADMIN_SCENARIO_ILLUSTRATION_REDESIGN.md` — UX/иллюстрации карточек сценария.
- `docs/ADMIN_SCENARIO_GRID_CHOICE.md` — сетка правой карточки.
- Прочие `docs/ADMIN_*.md` — не противоречить им по UX; визуал согласовать с этим мастер-планом.
- **`ADMIN_UI_EXTENSIONS.md`** — как добавлять новые экраны админки.

## 5. Инструкция для AI-агента

### 5.1 В начале сессии по админке

1. Прочитать **этот файл** (`docs/ADMIN_DODO_ENGINEERING_REDESIGN.md`).
2. Проверить **текущую фазу** (спросить пользователя или смотреть прогресс в чеклисте).
3. Не трогать графики и публичный дашборд без явного запроса.

### 5.2 Обязанности агента по направлению пользователя

- **Напоминать следующий шаг:** после Фазы 0 → Фаза 1 и т.д.
- **Напоминать про забытое:** Juneau/fallback, чеклист страниц, хардкод оранжевого в иллюстрациях админки.
- **Предлагать фиксировать решения** в этом документе (дата, выбранный accent, откат на A).
- При завершении фазы — кратко перечислить, что проверить вручную (скрин, контраст, reduced motion).

### 5.3 Если пользователь пишет без контекста

Коротко ответить, на какой мы фазе, и что логично сделать дальше по плану.

### На каждой фазе

- **Скрин или Loom** «то / не то» — быстрее, чем длинный текст.
- **Одна правка за раз** по приоритету (например: сначала контраст табов, потом карточки).
- Крупные решения — **одной строкой в журнал** (§6), чтобы следующий чат не спорил с прошлым.

- Экспорт **точных hex** из Figma брендбука, если отличаются от слайдов.
