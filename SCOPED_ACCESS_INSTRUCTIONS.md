# Инструкция: ограниченный доступ по юнитам и командам

Эта инструкция описывает, как включить и использовать возможность ограничивать доступ пользователей по юнитам и командам в админке.

---

## Что должно быть уже сделано

Перед применением этих шагов у вас должны быть:

- Таблицы из **`supabase-schema.sql`** (initiatives, people, person_initiative_assignments и т.д.).
- Таблица **`allowed_users`** и политики доступа из **`supabase-migration-access-control.sql`** (или эквивалентные миграции).
- Желательно: исправление рекурсии RLS из **`supabase-fix-rls-recursion.sql`** (функции `current_user_has_access` и `current_user_is_allowed_admin`). Если не применяли — не страшно: первая миграция ниже создаёт `current_user_has_access()` заново.

---

## Шаг 1. Применить миграции в Supabase

Есть два варианта: через CLI или вручную через SQL Editor.

### Вариант A: через Supabase CLI (если используете)

```bash
# Из корня проекта
npx supabase db push
```

Миграции выполнятся в порядке имён файлов: сначала `20260227120000_scoped_access_allowed_users.sql`, затем `20260227120001_scoped_access_rls.sql`.

### Вариант B: вручную в Supabase Dashboard

1. Откройте [Supabase Dashboard](https://supabase.com/dashboard) → ваш проект → **SQL Editor**.
2. **Первая миграция:**
   - Создайте новый запрос.
   - Скопируйте **весь** код из файла:
     - **`supabase/migrations/20260227120000_scoped_access_allowed_users.sql`**
   - Нажмите **Run**.
   - Убедитесь, что запрос выполнился без ошибок (в таблице `allowed_users` появятся колонки `allowed_units` и `allowed_team_pairs`).
3. **Вторая миграция:**
   - Новый запрос.
   - Скопируйте **весь** код из файла:
     - **`supabase/migrations/20260227120001_scoped_access_rls.sql`**
   - Нажмите **Run**.
   - Убедитесь, что выполнилось без ошибок.

## Шаг 2. Проверить схему (по желанию)

В Supabase: **Table Editor** → таблица **`allowed_users`**.

Должны быть колонки:

- `id`, `email`, `role`, `created_at` — как раньше;
- **`allowed_units`** — тип `text[]`, по умолчанию `{}`;
- **`allowed_team_pairs`** — тип `jsonb`, по умолчанию `[]`.

У уже существующих пользователей эти поля будут пустыми — для них по-прежнему «полный доступ».

## Шаг 4. Настроить ограниченный доступ в админке

1. Перейдите в раздел **Управление** (доступен только админам).
2. Откройте вкладку **Доступ** (список пользователей с доступом).
3. В таблице появилась колонка **«Доступ»**:
   - У **админов** отображается: **«Всё (админ)»** — настройка scope для них не используется.
   - У **пользователей** (роль «Пользователь»): показывается **«Всё»** или **«Юниты: N, Команды: M»** и кнопка с иконкой шестерёнки (**Настроить**).
4. Чтобы ограничить пользователя:
   - Нажмите **Настроить** (иконка шестерёнки) в строке нужного пользователя.
   - Откроется диалог **«Ограничить доступ»**.
   - **Полный доступ:** если галочка **«Полный доступ (видеть все юниты и команды)»** включена — пользователь видит всё (как раньше). Оставьте так для пользователей без ограничений.
   - **Ограничение:** снимите галочку «Полный доступ». Появятся блоки:
     - **Юниты** — отметьте те юниты, которые пользователь может видеть.
     - **Команды (юнит → команда)** — отметьте конкретные пары «юнит → команда».
   - Пользователь увидит только данные, попадающие в выбранные юниты и/или пары (юнит, команда). Можно выбрать только юниты, только команды или и то, и другое.
   - Если сняли «Полный доступ» и ничего не выбрали — пользователь не будет видеть данные (в диалоге есть предупреждение).
5. Нажмите **Сохранить**. Изменения применяются сразу; при следующем входе или обновлении страницы этот пользователь увидит только разрешённые юниты и команды.

---

## Шаг 5. Проверить поведение

1. **Под админом:** убедитесь, что дашборд и все данные по-прежнему видны.
2. **Под ограниченным пользователем:**
   - Добавьте тестового пользователя с ролью «Пользователь» и ограничьте его, например, одним юнитом или одной парой (юнит + команда).
   - Выйдите из админа и войдите под этим пользователем (Google с нужным @dodobrands.io).
   - На дашборде (Budget, Stakeholders, Timeline и т.д.) должны отображаться только инициативы/люди из выбранных юнитов и команд.
3. В раздел **Управление** ограниченный пользователь заходить не может (доступ только у админов).

---

## Логика доступа (напоминание)

- **Админ** — всегда видит всё, ограничения по юнитам/командам не применяются.
- **Пользователь с пустыми списками** (`allowed_units` и `allowed_team_pairs` пустые) — полный доступ (как раньше).
- **Пользователь с непустыми списками** — видит только:
  - инициативы/людей в выбранных **юнитах**;
  - и/или инициативы/людей в выбранных парах **(юнит, команда)**.
- Одна и та же команда в разных юнитах считается разными парами (выбирается именно пара «юнит → команда»).

## Если что-то пошло не так

- **Ошибка при Run миграции:** убедитесь, что таблица `allowed_users` уже создана (миграция доступа) и что скопирован весь текст файла без обрезки.
- **Пользователь с ограничением видит всё:** проверьте, что у пользователя роль `user` (не `admin`) и что в диалоге снята галочка «Полный доступ» и сохранены нужные юниты/команды. Обновите страницу или перелогиньтесь.
- **Пользователь не видит ничего:** возможно, у него задано ограничение, но не отмечен ни один юнит и ни одна команда — откройте «Настроить» и либо отметьте нужные юниты/команды, либо включите «Полный доступ» и сохраните.
- **В диалоге пустой список юнитов/команд:** списки берутся из данных в таблицах `initiatives` и `people`. Если там ещё нет записей с заполненными `unit`/`team`, списки будут пустыми. Добавьте хотя бы одну инициативу или человека с указанными юнитом и командой, затем обновите страницу «Доступ» и снова откройте «Настроить».
